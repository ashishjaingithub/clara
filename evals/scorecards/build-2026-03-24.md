## What Needs Work

1. **startup.ts has no test coverage (highest priority).** The production environment enforcement
   function (`checkProductionEnv`) calls `process.exit(1)` if required env vars are absent, but
   there is no test that verifies this behavior. A refactor could silently drop a required var
   check. Chaos agent provided a complete test template — this should be the first task for
   the next QA session. File: `src/__tests__/unit/lib/startup.test.ts`.

2. **No route handler integration tests.** Unit tests cover auth, rate limiting, and the
   receptionist in isolation. The composition of auth middleware + rate limiter + DB + agent
   in each route handler is untested. This is acceptable for Explore phase but is a blocker
   for Extract promotion. Routes needing tests: /api/chat, /api/demo, /api/leads,
   /api/admin/cleanup.

3. **Railway IP extraction test is directionally ambiguous.** The rate-limit test for
   `x-forwarded-for` parsing describes "leftmost public IP" but the implementation uses
   rightmost-non-private (Railway trusted proxy behavior). The test input happens to produce
   the same result either way. A targeted test case distinguishing leftmost-spoofed vs.
   rightmost-trusted should be added to prevent a future regression.

4. **ChatArea.tsx and WelcomeMessage.tsx not confirmed present.** FE-001 spec called for 8
   components; 7 are in the directory listing. These may be inlined in DemoPageClient. Verify
   and either extract or document the deviation.

5. **console.log in migrate.ts** is benign in production (runs once at startup) but will appear
   in Railway logs on every deploy. Consider replacing with structured logging or suppressing
   behind a `verbose` flag.
